Command Line: PDL Configuration

These command line options control how the GUI communicates with PDL when it sends products. For example, the following command line can be used to tell the GUI to send products to the PDL production system. 

java -jar AftershockGUI-2026_01_06.jar --pdl=prod --privateKey=mykeyfile

In this example, the --pdl option specifies that products should be sent to the production PDL system, and the --privateKey option specifies the digital key file that is used to sign products. Note that the name of the GUI jar file may be different than shown.

PDL configuration options are described in detail below. All PDL configuration options are optional; if omitted, then the GUI supplies defaults.

Options may appear in the command line in any order.

 

--pdl=dryrun
--pdl=dev
--pdl=prod

The --pdl option selects the destination when sending PDL products. It can have the following values:

  • dryrun - Nothing is sent to PDL, but the GUI "goes through the motions" of sending products.
  • dev - Send products to the PDL staging system.
  • prod - Send products to the PDL production system.

Note that not all versions of the GUI are able to send products to the PDL staging system. If the GUI does not support sending products to PDL staging, then dev effectively behaves the same as dryrun.

if the --pdl option is omitted, the default is dryrun.

 

--privateKey=PRIVATEKEYFILE

The --privateKey option specifies the name of a file that contains a digital key. The key is used to digitally sign products.

If the --privateKey option is omitted, the default is to send products unsigned.

The PDL production system requires that products be signed. Older versions of the PDL staging system accepted unsigned products, but newer versions require that products be signed.

 

--target=socket
--target=aws
--target=both

The --target option selects the PDL hub and communication protocal used when sending PDL products. It can have the following values:

  • socket - Send products to the on-premise PDL hubs using the PDL socket protocol. This is the old way to send products.
  • aws - Send products to the cloud PDL hubs using the PDL AWS protocol. This is the new way to send products.
  • both - Send products to both on-premise and cloud PDL hubs.

Note that because the GUI only supports a single key file, the both selection won't work unless both hubs are configured to accept the same digital key.

If the --target option is omitted, the GUI supplies a pre-programmed default.

 

--evseq-config=enable
--evseq-config=disable

The --evseq-config option specifies whether the GUI can generate and send event-sequence products. It can have the following values:

  • enable - The GUI can generate and send event-sequence products.
  • disable - The GUI cannot generate and send event-sequence products.

Note that if the OAF server does not support event-sequence products, you cannot make the server generate event-sequence products by enabling event-sequence in the GUI and then sending parameters to the server.

if the --evseq-config option is omitted, the GUI supplies a pre-programmed default.

 

--source=PRODUCTSOURCE

The --source option identifies the seismic network (or equivalent) that is the originator of products sent to PDL by the GUI. The default value is "us", which is the network code for the National Earthquake Information Center (NEIC). For example, --source=us.

Note that the digital key used to sign PDL products must be authorized for use with the specified seismic network.

 

--type=PRODUCTTYPE

The --type option specifies the type of PDL product that the GUI uses to send aftershock forecasts to PDL. The default value is "oaf", which uses the standard operational aftershock forecast product. For example, --type=oaf.

You can use "oaf-scenario" to send aftershock forecasts for an earthquake scenario exercise. For example, --type=oaf-scenario. There is no way to send an event-sequence product for an earthquake scenario, so if you use --type=oaf-scenario then you should also use --evseq-config=disable (this does not happen automatically).

Note that the digital key used to sign PDL products must be authorized for use with the specified type of product.
